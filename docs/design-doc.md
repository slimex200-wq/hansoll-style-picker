## Problem Statement

Hansoll Textile sends seasonal sample collection PDFs to Talbots Outlet buyers via email. Buyers review the PDF, then email back their style selections and feedback. This roundtrip is slow, static, and loses context — comments aren't attached to specific styles, selection status is tracked manually, and the Hansoll team has no real-time visibility into buyer preferences.

## Demand Evidence

Both sides of the workflow are affected:
- **Hansoll design/sales team** — must manually track which styles each buyer selected, compile email feedback, and reconcile conflicting responses across threads.
- **Talbots buyers** — must scroll through a static PDF, mentally note preferences, then compose a separate email describing which styles they want. No structured way to say "yes to this, maybe to that."

The current SP'27 collection has 5 styles (HDW127182, HDW127079, HDW126208, HMW320041, HMW326235). Each style has detailed fabric specs, multiple product photos, and fabric suggestions — information that's hard to reference in an email reply.

## Status Quo

1. Hansoll design team creates PDF with style photos, fabric specs, and suggestions
2. PDF is emailed to Talbots buyer(s) as attachment
3. Buyer opens PDF, reviews styles
4. Buyer composes email reply with selections and notes
5. Hansoll team reads email, manually tracks preferences
6. Back-and-forth emails for clarifications

**Cost:** Multiple email roundtrips per season. Context lost between threads. No structured data — everything is prose in email bodies.

## Target User & Narrowest Wedge

**Primary user:** Talbots Outlet buyer who receives Hansoll's seasonal collection PDF. Needs to quickly review styles on mobile, mark preferences (shortlist/maybe/pass), and leave specific notes per style.

**Secondary user:** Hansoll sales/design team member who needs to see buyer feedback in real-time without waiting for email replies.

**Narrowest wedge:** Mobile-first card grid with style photos/specs + structured selection (Shortlist/Maybe/Pass) + shared memos per style + admin summary view.

## Constraints

1. No authentication — link-only access with name prompt
2. Style data manually extracted from PDF (static JSON)
3. Mobile-first responsive design
4. Must work on modern browsers (Chrome, Safari, Edge)
5. Free-tier infrastructure (Supabase free, Vercel free)

### Accessibility (added by design review)

- Touch targets: 44px minimum (buttons, card tap area)
- Color contrast: all text meets WCAG 2.1 AA (4.5:1 for body, 3:1 for large text)
- Status badges: color + text label (not color-only — accessible to colorblind users)
- Drawer: trap focus when open, Escape to close, aria-modal="true"
- Keyboard nav: Tab through cards, Enter to open drawer, Escape to close
- Memo input: aria-label="Add a memo for style {ID}"
- Admin summary: semantic table markup, screen reader friendly

## Premises

1. Buyers access via link without login — name-only identification
2. Data is manually extracted from PDF into static JSON (auto PDF parsing unnecessary for 5 styles)
3. Shared persistence via Supabase from v1 — memos and selections visible to both sides
4. Mobile-first — buyers likely review on phones

## Cross-Model Perspective

Codex independently identified this as "a lightweight buyer-vendor review layer, not a digital lookbook" — sitting between PDF/email and PLM. Key insight: the product is "the smallest tool that collapses the feedback loop and keeps comments attached to the exact style being discussed."

Codex challenged the localStorage premise (#3), arguing that local-only memos defeat the core purpose — Hansoll can't see buyer feedback without shared persistence. This was accepted; v1 will use Supabase.

Codex recommended: Next.js + TypeScript + Tailwind + Supabase + Vercel, with signed collection URL + name prompt, structured actions (Shortlist/Maybe/Pass), shared notes, and a summary view. No login, no CMS, no notifications.

## Approaches Considered

### Approach A: Minimal Viable (CHOSEN)
- **Stack:** Next.js + Tailwind + Supabase + Vercel
- **Effort:** S (coding 2-3 hours + content prep 1 hour)
- **Risk:** Low
- **Features:** Mobile-first card grid, style detail drawer, Shortlist/Maybe/Pass, shared memos, name prompt, admin summary view
- **Pros:** Fastest to ship, free-tier infrastructure, proven stack
- **Cons:** Content management is manual (JSON + Supabase Storage)

### Approach B: Static Single Page
- **Stack:** Pure HTML/CSS/JS + Supabase
- **Effort:** S (1-2 hours)
- **Risk:** Low
- **Pros:** Minimal complexity, no build tools
- **Cons:** Hard to extend, manual HTML editing for new content

### Approach C: Full Platform
- **Stack:** Next.js + Tailwind + Supabase + Admin CMS
- **Effort:** M (1-2 days)
- **Risk:** Medium
- **Pros:** Reusable for future seasons, non-dev content management
- **Cons:** Over-engineered for current need

## Recommended Approach

**Approach A: Minimal Viable.** Fastest path to replacing the PDF+email workflow. Supabase provides shared persistence on free tier. Vercel provides zero-config deployment. The structure supports upgrading to Approach C later if the tool proves valuable across multiple seasons.

### Screens

1. **Name Prompt** — Modal overlay on first visit. "Enter your name" + "View Collection" button. Name stored in localStorage + sent with all Supabase writes.

2. **Card Grid (Main)** — SaveDay-style responsive grid. Each card: style photo, style ID, fabric summary, status badge (Shortlist/Maybe/Pass), memo count. Header: "HANSOLL SP'27" + progress indicator ("3/5 reviewed") + "Summary" link. Progress = styles where current user has set any selection status.

3. **Detail Drawer** — Bottom sheet on card tap. Photo gallery (swipe carousel: main photo → fabric closeup → detail shots), spec table (Fabric #, Contents, Construction, Weight, Designed by), fabric suggestion highlight box, action buttons (Shortlist/Maybe/Pass), memo thread (all memos visible to all users — buyers + Hansoll team share one thread per style). Cards show 1 photo only (3:4 aspect ratio, object-fit: cover); all additional photos in drawer gallery. Swipe to dismiss drawer on mobile.

4. **Admin Summary** — `/admin` route. List of all styles with vote counts, latest memo per style, review progress (% of users who have voted on at least one style). Hansoll team's dashboard. Style data lives in Supabase `styles` table, seeded once from manually-prepared JSON.

### Data Model (Supabase)

```
styles (static, seeded once)
  id: text (PK) — e.g. "HDW127182"
  collection: text — "SP27-TALBOTS-OUTLET"
  fabric_no: text
  contents: text
  construction: text
  weight: text
  finishing: text
  designed_by: text
  image_url: text
  fabric_suggestion: jsonb — { fabric_no, contents, weight }

selections
  id: uuid (PK)
  style_id: text (FK)
  collection: text — denormalized for RLS filtering
  user_id: text — localStorage UUID (anonymous identifier)
  user_name: text — display name (entered at name prompt)
  status: text — "shortlist" | "maybe" | "pass"
  created_at: timestamp
  updated_at: timestamp
  UNIQUE(style_id, user_id)

memos (append-only — no edit, no delete)
  id: uuid (PK)
  style_id: text (FK)
  collection: text — denormalized for RLS filtering
  user_id: text — localStorage UUID
  user_name: text — display name
  content: text
  created_at: timestamp
```

### RLS Policy (Supabase)

- `styles`: anon can SELECT where collection matches (read-only)
- `selections`: anon can SELECT, INSERT, UPDATE where style_id belongs to accessible collection. UPDATE limited to own user_id only.
- `memos`: anon can SELECT, INSERT where style_id belongs to accessible collection. No UPDATE/DELETE (append-only).

Note: selections와 memos에 `collection` 컬럼 추가 (denormalize) → RLS에서 직접 필터링.

### Tech Decisions

- **No auth:** Supabase anon key + RLS with collection filter. Link-only access. Each user gets a random UUID in localStorage as identifier; display name is separate.
- **Images:** Supabase Storage (free 1GB) + Next.js `<Image>` 컴포넌트 (자동 리사이즈/WebP 변환). `next.config.js`에 Supabase Storage domain을 `remotePatterns`에 추가.
- **Deployment:** Vercel free tier. Custom domain optional later.
- **Admin:** `/admin` route는 별도 인증 없음 — URL을 아는 사람만 접근 (v1에서는 보안 불필요, 민감 데이터 아님).
- **Realtime:** Admin summary 뷰만 Supabase realtime subscription. 카드 그리드는 page load 시 fetch + 사용자 액션 시 refetch.

### Testing Strategy (added by eng review)

**Error UI:** 모든 Supabase mutation (selection upsert, memo insert)에 try-catch + toast notification. 성공 시 체크 토스트, 실패 시 에러 토스트 ("저장 실패. 다시 시도해주세요.").

**Unit tests (Vitest + React Testing Library):**
- Supabase helper functions: selection upsert (INSERT + UPDATE), memo insert, empty memo rejection
- 컴포넌트 렌더링: StyleCard (with/without status badge), DetailDrawer (spec table), AdminSummary (vote aggregation)
- RLS 검증: anon key로 다른 collection 접근 시 빈 결과 반환 확인

**Skip for v1:** E2E tests, visual regression, performance benchmarks

### Error Handling (added by eng review)

```
Selection UPSERT 실패 → toast "선택 저장 실패" + 버튼 상태 롤백
Memo INSERT 실패    → toast "메모 저장 실패" + 입력 내용 유지 (사라지지 않음)
Styles fetch 실패   → 에러 화면 ("연결 실패. 새로고침 해주세요.")
Admin realtime drop → 자동 재연결 (Supabase 기본 동작) + "마지막 업데이트: N분 전" 표시
```

## Open Questions

1. Should the collection URL be guessable or use a random slug for mild privacy?
2. Do we need multiple collections at once, or is single-collection sufficient for v1?
3. Should memos support images/attachments (e.g., buyer sketching modifications)?
4. v1 known limitation: user identity is not verified (anon key + client-provided user_id). Buyer name spoofing is technically possible but acceptable given non-sensitive data and internal use.

## Success Criteria

1. Talbots buyer can review all 5 styles and leave selections+memos within 5 minutes on mobile
2. Hansoll team can see buyer feedback in real-time without email
3. Replaces at least one email roundtrip per collection review cycle
